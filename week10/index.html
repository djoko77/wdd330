Client-side Form Validation

  • When you enter data, the browser and/or the web server will check to see that the data is in the correct format
  • client-side validation is a validation done the browser
  • server-side validation is a validation done on the server
  • 3 reasons to validate the form data
    • We want to get the right data, in the right format
    • We want to protect our users' data
    • We want to protect ourselves
  • Types of Client-Side Validation
    • Built-in form validationusing HTML5 form validation
      • required (mandatory input)
      • minlength and maxlength (for string)
      • min and max (for numbers)
      • type (number, email address)
      • pattern (regular expression)
    • JavaScript validation
      • validationMessage returns a localized message describing the validation constraints that the control doesn't satisfy (if any)
      • validity returns a ValidityState object that contains several properties describing the validity state of the element
        • patternMismatch
        • tooLong
        • tooShort
        • rangeOverflow
        • rangeUnderflow
        • typeMismatch
        • valid
        • valueMissing
        • willValidate
      • checkValidity() returns true if the element's value has no validity problems)
      • reportValidity() reports invalid field(s) using events
      • setCustomValidity(message) adds a custom error message to the element

Using the Fetch API

  • The Fetch API provides a JavaScript interface for accessing and manipulating parts of the HTTP pipeline (responses and requests)
  • A fetch() promise will reject with a TypeError when a network error is encountered or CORS is misconfigured on the server-side
  • Response instances are returned when fetch() promises are resolved
    • Response.status is an integer with response status code (default value 200)
    • Response.statusText is an HTTP status code message (default value “”)
    • Response.ok checks the status (boolean))
